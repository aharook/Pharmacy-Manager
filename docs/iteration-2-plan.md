## Сценарії з Backlog, які реалізуються

### Основні сценарії
1. **Створення замовлення з перевіркою складу** (розповсюджена з Lab 34)
   - Перевірка наявності товару перед додаванням у замовлення
   - Застосування знижки на основі типу карти клієнта
   - Розрахунок підсумкової вартості

2. **Бронювання ресурсів** (новий сценарій)
   - Створення бронювання товарів на певний період
   - Перевірка конфліктів бронювань (один товар не може бути забронець одночасно)
   - Скасування бронювання та повернення товару в доступність
   - Перетворення бронювання в завершене замовлення

3. **Видача та повернення товару** (новий сценарій)
   - Відстеження видачі товару клієнту
   - Облік поверненої кількості та статусу
   - Перевірка терміну прострочення
   - Стягнення штрафу за прострочення

## Точки розширення (Extension Points)

### 1. **Strategy Pattern для знижок**
   - Уже присутня в Lab 34 через `IDiscountStrategy`
   - У Lab 35 розширимо для поддержки:
     - Комбінованих знижок (поєднання кількох стратегій)
     - Умовних знижок (знижка залежить від кількості або суми)
     - Часових знижок (знижка залежить від часу)

### 2. **Repository Pattern для персистентності**
   - Контракт `IInventoryRepository` вже присутня
   - Додаємо `IOrderRepository`, `IBookingRepository`
   - Дозволяє змінювати реалізацію (файл → БД) без зміни бізнес-логіки

### 3. **Observer Pattern для подій замовлення**
   - При зміні статусу замовлення спрацьовує подія
   - Дозволяє додавати нові обробники (логування, сповіщення) без зміни сервісу

### 4. **Factory Pattern для бронювань**
   - Централізоване створення бронювань з валідацією
   - Дозволяє додавати нові типи бронювань без змін контролера

## Ідентифіковані ризики

### Ризик 1: Дублювання логіки перевірки складу
**Проблема:** Логіка перевірки складу розповсюджена в:
- `InventoryService.reserveItems()`
- `PharmacyController.createOrder()`
- `BookingService.createBooking()`

**Як мінімізувати:** Виділити спільну логіку у окремий метод `validateStockAvailability()`

### Ризик 2: Конфлікти між бронюваннями і замовленнями
**Проблема:** Один товар може бути:
- У складі
- Забронький
- У замовленні
Требага точна синхронізація статусів.

**Як мінімізувати:**
- Явна трансакційність: бронювання → замовлення → видача
- Тести для перевірки конфліктних сценаріїв

### Ризик 3: JSON-персистентність не гарантує цілісність при збої
**Проблема:** Якщо сервер впаде під час запису, файл може бути пошкоджений.

**Як мінімізувати:**
- Запис у тимчасовий файл + atomic rename
- Backup-копія перед кожним оновленням
- Обробка помилок при завантаженні

### Ризик 4: Слабкі інтерфейси для нових сервісів
**Проблема:** BookingService і ReportService мають чітко визначені контракти?

**Як мінімізувати:**
- Визначити інтерфейси (IBookingService, IReportService) з першого дня
- Приховати реалізацію за контрактом

### Ризик 5: Складність тестування персистентності
**Проблема:** FileInventoryRepository залежить від файлової системи.

**Як мінімізувати:**
- Мокування файлової системи через інтерфейс IFileStorage
- Або використання тимчасових директорій для тестів
